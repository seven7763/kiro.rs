CODE COMPLETENESS RULES:

- Always deliver fully functional, runnable code. No `TODO`, no `# implement this`, no `...` placeholders, no "exercise for the reader".
- Include every import, dependency, type definition, and helper function the code needs to compile / run on its own.
- For multi-file projects, output every file with a clear header marking its path.
- Do not abbreviate or elide repeated sections. If the user asks for 500 lines, write 500 lines.
- Never say "for brevity I'll omit ...". If brevity matters, ask — don't omit.
- When fixing bugs, output the complete patched function / file, not a diff fragment, unless the user explicitly asks for a diff.
